════════════════════════════════════════════
METADATA EXTRACTION SQL
════════════════════════════════════════════

If metadata is not yet provided, generate and return these queries first:

ORACLE (run as <ORACLE_SCHEMA> or DBA):
  SELECT t.table_name, c.column_name, c.data_type,
         c.data_length, c.data_precision, c.data_scale,
         c.nullable, c.column_id,
         cc.comments AS column_comment
  FROM   all_tables t
  JOIN   all_tab_columns c ON c.owner = t.owner AND c.table_name = t.table_name
  LEFT   JOIN all_col_comments cc ON cc.owner = c.owner
         AND cc.table_name = c.table_name AND cc.column_name = c.column_name
  WHERE  t.owner = UPPER('<ORACLE_SCHEMA>')
  ORDER  BY t.table_name, c.column_id;

SNOWFLAKE (run in <SF_DATABASE.SCHEMA>):
  SELECT table_name, column_name, data_type,
         character_maximum_length, numeric_precision, numeric_scale,
         is_nullable, ordinal_position, comment
  FROM   information_schema.columns
  WHERE  table_schema = UPPER('SCHEMA>')
  ORDER  BY table_name, ordinal_position;

COLUMN MATCHING METHODOLOGY
════════════════════════════════════════════

Apply ALL of the following matching strategies in priority order. When multiple strategies fire, combine their signals into a single confidence score (0–100%).

  1. Exact name match
  2. Normalised match (strip prefixes/suffixes)
  3. Abbreviation expansion (CUST→CUSTOMER)
  4. CamelCase / snake_case alignment
  5. Edit-distance fuzzy match
  6. Semantic synonym inference
  7. Domain-aware synonym dictionary
  8. Sample data value overlap
  9. Enumeration / code set matching
  10. Column comment / description match
  11. Data type compatibility
  12. Length / precision alignment
  13. Primary key / FK relationship
  14. Nullability consistency

Priority order when strategies conflict:
  1. Exact name match always wins outright (100%).
  2. Normalised / abbreviation matches score 85–95% depending on edit distance.
  3. Semantic inference is capped at 80% without corroboration from data overlap.
  4. Data value overlap alone (no name similarity) is capped at 70%.
  5. Type compatibility is a modifier (+5% boost or –10% penalty), not a primary signal.

════════════════════════════════════════════
SEMANTIC MATCHING RULES
════════════════════════════════════════════

When column names differ but semantically refer to the same concept, you MUST infer equivalence. Key rules:

  1. PREFIX/SUFFIX STRIPPING: Oracle columns often carry table-prefix conventions (e.g., CUST_ID → CUSTOMER_ID, ACCT_NO → ACCOUNT_NUMBER). Strip these systematically.

  2. ABBREVIATION DICTIONARY: Expand common DWH abbreviations:
     AMT→AMOUNT, QTY→QUANTITY, DT→DATE, CD→CODE, DESC→DESCRIPTION,
     NBR/NO→NUMBER, TYP→TYPE, IND→INDICATOR/FLAG, ADDR→ADDRESS,
     CUST→CUSTOMER, ACCT→ACCOUNT, TXN→TRANSACTION, BAL→BALANCE,
     EFF→EFFECTIVE, EXP→EXPIRY, SRC→SOURCE, TGT→TARGET,
     STR→START, END→END, CRT→CREATED, UPD→UPDATED, DEL→DELETED,
     FLG→FLAG, SEQ→SEQUENCE, REF→REFERENCE, CAT→CATEGORY.
     Extend with domain-specific terms for enterprise data warehouse.

  3. SYNONYM INFERENCE: Treat the following as equivalent unless context contradicts:
     • CREATED_DATE / CREATION_DATE / CREATE_DT / INSERT_DATE / LOAD_DATE
     • MODIFIED_DATE / UPDATE_DATE / LAST_UPDATED / LAST_MODIFIED_DT
     • STATUS / STATE / STATUS_CD / RECORD_STATUS
     • EFFECTIVE_DATE / EFF_DT / START_DATE / VALID_FROM
     • EXPIRY_DATE / EXP_DT / END_DATE / VALID_TO
     • CUSTOMER_ID / CUST_ID / CLIENT_ID / PARTY_ID (in customer-facing domains)
     • ACCOUNT_ID / ACCT_ID / ACCT_NO / ACCOUNT_NUMBER

  4. DATA SAMPLE MATCHING: If sample rows are provided, compare value distributions (distinct count, min/max, format patterns like date formats, ID formats, code values) to corroborate or contradict a name-based match.

  5. ENUMERATION MATCHING: If two columns have the same small set of distinct values (e.g., both contain 'Y'/'N', or the same status codes), treat this as strong corroborating evidence even if names differ.

  6. COLUMN COMMENT MATCHING: If Oracle column comments describe the same business concept as the Snowflake column name or comment, treat this as high-confidence semantic evidence.

CONFIDENCE CLASSIFICATION
════════════════════════════════════════════

Classify every mapped pair using these thresholds:

  HIGH     ≥ 90%  — use directly in automated migration scripts.
  MEDIUM   65–89%  — include in scripts with a TODO comment for human verification.
  LOW      40–64%  — flag for mandatory manual review before scripting.
  UNMATCHED < 40%  — no match found; classify as ORACLE_ONLY or SNOWFLAKE_ONLY.

TABLE-LEVEL MATCHING

  • Document table-level confidence alongside column-level confidence.
  • If Oracle Informatica ETL merged multiple source tables into one target, note this as a MERGE pattern.
