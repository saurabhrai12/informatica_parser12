INFORMATICA TRANSFORMATION → SNOWFLAKE REWRITE RULES
════════════════════════════════════════════

Apply these exact translation rules for every transformation encountered in the lineage:

EXPRESSIONS & LOGIC:
  • IIF(cond, true_val, false_val)          → CASE WHEN cond THEN true_val ELSE false_val END
    Nested IIF                             → Nested CASE WHEN (no limit in Snowflake)
  • DECODE(col, v1,r1, v2,r2, default)      → CASE col WHEN v1 THEN r1 WHEN v2 THEN r2 ELSE default END
  • NVL(col, default)                        → COALESCE(col, default)
  • NVL2(col, not_null_val, null_val)        → IFF(col IS NOT NULL, not_null_val, null_val)
  • SUBSTR(str, pos, len)                    → SUBSTRING(str, pos, len)
  • INSTR(str, substr, pos, occurrence)      → POSITION(substr IN str) [simple] or REGEXP_INSTR for occurrence > 1
  • TO_DATE(str, fmt)                        → TO_DATE(str, fmt) [same syntax]
  • TO_CHAR(date, fmt)                       → TO_VARCHAR(date, fmt)
  • Oracle date arithmetic (date + n)        → DATEADD(day, n, date)
  • SYSDATE                                  → CURRENT_TIMESTAMP() or CURRENT_DATE()
  • Aggregator GROUP BY ports                → Standard SQL GROUP BY in CTE or subquery
  • Aggregator aggregate expression          → SUM/AVG/COUNT/MIN/MAX window or inline
  • Custom expression port                   → Inline SQL expression in SELECT clause with alias matching target column name

LOOKUPS & JOINS:
  • Connected Lookup (return port wired in)  → LEFT JOIN on lookup table with matching condition
    Lookup condition: LKP.key = input_port → LEFT JOIN lkp_table LKP ON LKP.key = src.col
  • Unconnected Lookup (:LKP.proc_name)     → Scalar subquery or LEFT JOIN; prefer LEFT JOIN for performance
    :LKP.LOOKUP_PROC(key) = val            → (SELECT return_col FROM lkp_table WHERE key_col = src.col LIMIT 1)
  • Joiner (Master + Detail)                 → Explicit JOIN type matching Joiner condition (INNER/LEFT/FULL)
    Normal join (both sorted)              → INNER JOIN
    Master outer join                      → RIGHT JOIN (detail is driving)
  • SQ override SQL                          → Reproduce verbatim as a CTE or subquery, then translate Oracle functions to Snowflake equivalents

FLOW CONTROL:
  • Router groups                            → Split the INSERT into separate INSERT INTO ... SELECT ... WHERE <group_condition> statements, one per Router group
    DEFAULT group                          → Final INSERT with NOT (group1_cond OR group2_cond ...)
  • Filter transformation                    → WHERE clause in the SELECT driving the INSERT/MERGE
  • Sorter                                   → ORDER BY in SELECT; or document in comment if not required for correctness (Snowflake sets are unordered)
  • Normalizer                               → UNPIVOT or LATERAL FLATTEN to expand repeated columns to rows
  • Sequence generator (CURRVAL/NEXTVAL)     → Create a Snowflake SEQUENCE object, reference with <SEQ_NAME>.NEXTVAL in the INSERT
  • Informatica stored procedure call        → CALL <procedure_name>(<args>) inside the Snowflake stored proc body

WRITE STRATEGY:
  • DD_INSERT only                           → INSERT INTO target SELECT ... FROM source WHERE <filter>
  • DD_UPDATE only                           → UPDATE target SET col = val FROM source WHERE target.key = source.key
  • DD_INSERT + DD_UPDATE (UPSERT)           → MERGE INTO target USING source ON (target.key = source.key)
                                                WHEN MATCHED THEN UPDATE SET ...
                                                WHEN NOT MATCHED THEN INSERT (cols) VALUES (vals)

  • DD_REJECT                                → Route to MIGRATION_ERROR_LOG with original row + error code + timestamp
  • Truncate + reload (truncate_flag = true) → TRUNCATE TABLE target; then INSERT INTO ... SELECT ...
    Wrap truncate + insert in same transaction block for atomicity

PROCEDURE STRUCTURE
════════════════════════════════════════════

Generate one stored procedure per Informatica mapping (one target table = one procedure as a default; flag if a mapping loads multiple targets and split accordingly).

Procedure signature pattern:
  CREATE OR REPLACE PROCEDURE <SF_DATABASE>.<TARGET_SCHEMA>.SP_<TARGET_TABLE_NAME>(
    -- Parameters defined here based on features enabled below
  )
  RETURNS VARCHAR
  LANGUAGE SQL
  ...

Structure each procedure body in this order:
  1. PRE-HOOK — execute any pre-SQL statements (e.g. disable constraints, set session params)
  2. IDEMPOTENCY CHECK — use a deduplication guard or DELETE-before-INSERT on key columns to ensure safe re-run
  3. INCREMENTAL FILTER — apply watermark (e.g. WHERE LAST_MODIFIED_DT > :last_run_ts) derived from MIGRATION_AUDIT_LOG
  4. MAIN TRANSFORMATION LOGIC — CTEs for each major transformation stage, then final INSERT / MERGE / UPDATE
  5. ROW COUNT VALIDATION — compare source count vs target rows affected; raise error if delta exceeds threshold
  6. EXCEPTION HANDLING — catch all errors, write to MIGRATION_ERROR_LOG, re-raise or return error status
  7. POST-HOOK — execute post-SQL statements (e.g. refresh materialized views, update statistics)

CTE PATTERN — MANDATORY
════════════════════════════════════════════

Translate multi-transformation chains into a WITH (CTE) ladder, one CTE per logical stage:

  WITH
  SQ_<MAPPING_NAME> AS (
    -- Source qualifier: reproduce SQ override SQL or base SELECT from source table
    -- Apply source filter from lineage
    SELECT <source columns mapped per lineage>
    FROM   <SOURCE_SCHEMA>.<source_table>
    WHERE  <source_filter_condition>
  ),
  EXPR_<MAPPING_NAME> AS (
    -- Expression transformer: apply all expression port rewrites
    SELECT
      <translated expressions as target column aliases>,
      ...
    FROM SQ_<MAPPING_NAME>
  ),
  LKP_<MAPPING_NAME> AS (
    -- Lookup(s): LEFT JOINs added here
    SELECT e.*, lkp.<return_col>
    FROM   EXPR_<MAPPING_NAME> e
    LEFT   JOIN <SF_DATABASE>.<TARGET_SCHEMA>.<lookup_table> lkp ON lkp.<lkp_key> = e.<input_port>
  )
  -- Final write: INSERT / MERGE using the last CTE
  MERGE INTO <SF_DATABASE>.<TARGET_SCHEMA>.<TARGET_TABLE> ...

Name each CTE after the Informatica transformation component it replaces. Add a comment on each CTE referencing the original Informatica transformation name and type.
